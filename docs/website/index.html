openAIOS

Model-agnostic AI agent orchestration OS. Connect any model to any channel with built-in governance, budget tracking, session management, and a governed inter-agent bus.

Model-agnostic Deny-by-default Docker isolation Agent bus Budget controls

What is openAIOS

An operating system layer for AI agents — not a framework you code in, but a runtime you configure.

openAIOS is a model-agnostic AI agent orchestration OS. It connects messaging channels (Telegram, Discord, Webhook) to AI runners (Claude Code, Ollama, OpenAI-compatible APIs) through a typed interface layer. You describe agents in a YAML config file; openAIOS handles all the plumbing: message routing, session continuity, governance checks, budget enforcement, and inter-agent communication.

Every agent operates under a deny-by-default permission model — the agent can only use the tools you explicitly allow. Budget limits cap spending per period and can automatically downgrade the model or block execution when limits are hit. The internal agent bus lets agents call other agents, with both layers (governance policy and callee allowlist) required to pass before a cross-agent call goes through.

In Docker mode, each agent runs in its own long-lived container with configurable memory and CPU limits, an isolated workspace volume, and optional capability sidecars (Chromium for browser automation). Turns are dispatched via docker exec, so there is no per-turn container startup overhead while the agent still enjoys strong OS-level isolation from the host and other agents.

Channels
Telegram, Discord (stub), Webhook HTTP
Runners
Claude Code CLI, Ollama, OpenAI-compat, Docker
Governance
LocalGovernance (in-process) or BRGovernance (external API)
Budget
SQLite-backed, per-agent, daily / weekly / monthly periods

Quickstart

Local dev with Ollama — no Telegram token, no cloud account required.

Prerequisites: Node.js 22+, pnpm, Ollama running locally with a model pulled.

  1. 1

    Clone and build

    bash
    git clone https://github.com/NeoForgeEngineering/openaios
cd openaios
pnpm install && pnpm build
  2. 2

    Create openAIOS.yml — an Ollama-backed agent on the webhook channel

    yaml
    agents:
  - name: assistant
    persona: "You are a helpful assistant. Be concise and accurate."
    model:
      default: ollama/qwen2.5-coder:7b
    channels:
      webhook:
        path: /webhook
    permissions:
      allow: [Read, Glob, Grep]
      deny: []
    runner:
      mode: native
    capabilities:
      browser: false
      agent-calls: []

models:
  providers:
    ollama:
      base_url: "http://localhost:11434"

network:
  bind: localhost
  port: 3000

data:
  dir: ./data
  3. 3

    Start the runtime

    bash
    node packages/cli/dist/bin/openaios.js start

    Expected output:

    log
    [openaios] Config loaded — 1 agent(s)
[openaios] Webhook server listening on http://localhost:3000
[openaios] Agent "assistant" ready (native, ollama/qwen2.5-coder:7b)
[openaios] RouterCore started
  4. 4

    Send a test message

    bash
    curl -s -X POST http://localhost:3000/webhook \
  -H 'Content-Type: application/json' \
  -d '{"text": "What is 2 + 2?", "userId": "me"}'

    Response:

    json
    {"output":"4","messageId":"msg_01ab..."}

Installation

Install openAIOS on any Linux or macOS machine in one command. No root required by default.

One-liner installer

bash
curl -fsSL https://raw.githubusercontent.com/NeoForgeEngineering/openaios/main/install.sh | bash

The installer handles everything automatically:

  1. Detects macOS / Linux
  2. Ensures Node.js 22+ — installs via nvm if not present, patches your shell profile
  3. Ensures pnpm — installs globally via npm if missing
  4. Clones the repo into ~/.openaios (or $OPENAIOS_DIR)
  5. Runs pnpm install && pnpm build
  6. Rebuilds native modules (better-sqlite3) for the running Node version
  7. Writes an openaios wrapper script to ~/.local/bin/openaios

After install, the full flow is:

bash
openaios init              # scaffold openAIOS.yml in current directory
$EDITOR openAIOS.yml       # configure your agents and channels
openaios start             # run in the foreground (Ctrl+C to stop)
openaios service install   # register as a background daemon
i

To install to a custom location, set OPENAIOS_DIR before piping:

bash
OPENAIOS_DIR=/opt/openaios curl -fsSL .../install.sh | bash

Service daemon

Register openAIOS as a long-running background service. Run this from the directory containing your openAIOS.yml:

bash
openaios service install

On Linux this installs a user-level systemd unit (~/.config/systemd/user/openaios.service) — no sudo needed. Pass --system for a system-level unit.

On macOS this writes a LaunchAgent plist to ~/Library/LaunchAgents/dev.openaios.agent.plist and loads it immediately.

Upgrading

Pull the latest release, rebuild, and restart the service in one command:

bash
openaios upgrade

This runs: git reset --hard origin/main → pnpm install → pnpm build → rebuild native modules → restart service (if registered).

How It Works

Every inbound message travels through a fixed pipeline: channel adapter → RouterCore → runner → channel adapter.

Normal turn flow

Telegram / Discord / Webhook ↓ InboundMessage ChannelAdapter.onMessage() ↓ RouterCore ├── SessionStore.get() → load session (claudeSessionId for --resume) ├── BudgetManager.check() → may downgrade model or block turn ├── RunnerAdapter.run() → execute turn (spawn claude / docker exec / HTTP) │ └── (docker) docker exec openaios-{name} claude [args...] ├── SessionStore.set() → persist updated session ├── BudgetManager.record() → record cost └── GovernanceAdapter.report() → fire-and-forget audit (BRGovernance) ↓ RunResult.output ChannelAdapter.send() ↓ Telegram / Discord / Webhook response

Agent-to-agent bus call

call_agent script (inside container) ↓ POST /internal/bus/message ↓ Authorization: Bearer {one-time UUID token} BusHttpServer (127.0.0.1:{random_port}) ↓ AgentBus.request() ├── GovernanceAdapter.checkPolicy(tool: 'call_agent') ├── allowedCallees check → callee in capabilities['agent-calls']? ├── BudgetManager.check(toAgent) ├── SessionStore.get(toAgent, callerSession) ├── RunnerAdapter.run(toAgent) ├── SessionStore.set() └── BudgetManager.record(toAgent) ↓ AgentBusResponse { output } call_agent prints output to stdout

Sessions

The SessionStore persists the claudeSessionId returned by each Claude Code turn and passes it back via --resume on the next turn. In Docker mode, the long-lived container retains conversation history in-process across turns without needing --resume, but the session ID is still tracked for compatibility. Ollama and OpenAI-compatible runners maintain conversation history in-memory (keyed by session key), which is lost on restart.

Startup sequence

log
openaios start
  1.  Load config (openAIOS.yml, resolve ${ENV_VAR} references)
  2.  Init: SessionStore, BudgetManager, Governance
  3.  If any webhook agents → start shared HTTP server on network.port
  4.  Start bus HTTP server (127.0.0.1:{random_port}, one-time token)
  5.  If any docker agents → create ContainerOrchestrator + CapabilityProvisioner
  6.  For each docker agent → ensureRunning() (docker run if not already up)
  7.  For each docker agent → provision() (start browser sidecar if needed)
  8.  For each agent → register on AgentBus
  9.  For each agent channel → create adapter, push AgentRoute
 10.  RouterCore.start() → all channels begin polling/listening
 11.  On SIGINT/SIGTERM → stop channels → close servers → deprovision → stop containers

Channels

Channels are the ingress/egress points for agent messages.

Webhook

The webhook channel uses a synchronous request/response pattern: the HTTP POST is held open until the agent finishes, then the response is returned in the HTTP body. This makes curl a first-class local dev client — no Telegram account required.

Multiple agents can share the same HTTP server by registering different paths. The server starts only if at least one agent has channels.webhook configured.

Request

bash
curl -X POST http://localhost:{port}{path} \
  -H 'Content-Type: application/json' \
  [-H 'X-Webhook-Secret: {secret}'] \
  -d '{"text": "...", "userId": "optional", "messageId": "optional"}'

Response

json
{"output": "...", "messageId": "..."}

Config

yaml
channels:
  webhook:
    path: /webhook          # URL path for this agent
    secret: ${WEBHOOK_SECRET}  # Optional — validated via X-Webhook-Secret header

Multi-agent path routing: give each agent a distinct path (/webhook/assistant, /webhook/coder) and they all share port 3000.

Telegram

The Telegram adapter uses outbound long-polling — no inbound ports are required. Obtain a bot token from @BotFather and set it in your environment:

yaml
channels:
  telegram:
    token: ${TELEGRAM_TOKEN}   # From @BotFather
✓

Because Telegram uses polling (not inbound webhooks), agents on Tailscale-only networks receive messages without any port-forwarding or public IP.

Discord Coming soon

The Discord adapter is stubbed. A bot token and guild ID will be required. The config key is channels.discord.token / channels.discord.guildId.

Runner Modes

The runner is selected by the runner.mode field and/or the model name prefix.

native

Spawns a new process per turn directly on the host using the claude CLI. Requires Claude Code CLI in PATH (or set models.providers.claude-code.bin). Suitable for development and single-agent deployments where container overhead is not desired.

yaml
runner:
  mode: native
log
openaios daemon
  └── spawn('claude', [...flags, '--', userMessage])  ← per turn
!

Isolation note: In native mode, all agents run as the same OS user. A compromised agent can read any file accessible to that user. Use Docker mode for stronger isolation.

docker

Each agent gets a long-lived container (openaios-{name}) started once at startup. Turns execute via docker exec — persistent workspace via a named volume, configurable memory and CPU limits, no per-turn container startup cost.

yaml
runner:
  mode: docker
  docker:
    image: openaios/agent:latest   # image with claude CLI + call_agent tool
    memory: 1g                     # container memory limit
    cpus: 1                        # CPU limit
log
openaios daemon
  ├── ContainerOrchestrator
  │    └── docker run tail -f /dev/null   ← started once at startup
  └── per turn: docker exec openaios-{name} claude [args...]

Container naming:

ResourceName
Agent containeropenaios-{agentName}
Browser sidecaropenaios-{agentName}-browser
Docker networkopenaios (bridge, created idempotently)
Workspace volumeopenaios-{agentName}-workspace

Ollama

Prefix the model name with ollama/ to route through the Ollama local API. Configure the base URL under models.providers.ollama.

yaml
model:
  default: ollama/qwen2.5-coder:7b   # prefix selects Ollama runner

models:
  providers:
    ollama:
      base_url: "http://localhost:11434"

Conversation history is maintained in-memory per session. It is lost on process restart. No API key is required for local Ollama.

OpenAI-compatible

Prefix the model with openai/ to route through any OpenAI-compatible API endpoint. Works with Groq, Together, OpenRouter, local vLLM/llama.cpp servers, and the official OpenAI API.

yaml
model:
  default: openai/gpt-4o-mini

models:
  providers:
    openai:
      api_key: ${OPENAI_API_KEY}
      base_url: "https://api.openai.com/v1"  # optional override

    openrouter:
      api_key: ${OPENROUTER_API_KEY}
      base_url: "https://openrouter.ai/api/v1"
Model prefixRunner
claude-codeClaude Code CLI (local, native mode)
ollama/...Ollama local API

Configuration Reference

All configuration lives in openAIOS.yml. Use ${ENV_VAR} references for secrets — never inline credentials.

agents

An array of agent definitions. At least one agent is required.

    persona: "You are a helpful assistant."  # Inline persona or path to CLAUDE.md
    model:

      premium: claude-code        # Optional — escalated model (future feature)
    channels:
      telegram:
        token: ${TELEGRAM_TOKEN}
      webhook:
        path: /webhook

      deny: [Bash, WebSearch]           # Takes precedence over allow

Agent Bus

Governed request/response between agents — Agent A calls Agent B within enforced policy bounds.

The agent bus is an internal HTTP server (127.0.0.1:{random_port}) that routes cross-agent calls. Inside a Docker container, agents invoke the call_agent bash script, which POSTs to the bus endpoint. The bus applies two independent authorization layers before dispatching the call to the target agent's runner.

Two-layer authorization

LayerCheckWhere
Layer 1 GovernanceAdapter.checkPolicy({ tool: 'call_agent' }) — call_agent must be in the calling agent's permissions.allow list (auto-added when capabilities['agent-calls'] is non-empty) AgentBus
Layer 2 The target agent must be listed in the caller's capabilities['agent-calls'] (allowedCallees check) AgentBus

Both layers must pass. This prevents agents calling peers they were not explicitly given permission to reach, and prevents circular delegation chains from bypassing governance (each hop is independently checked).

Example: Agent A calls Agent B

yaml
agents:
  - name: orchestrator
    persona: "You are a planning agent. Delegate coding tasks to the coder agent."
    model:
      default: ollama/qwen2.5:7b
    channels:
      webhook:
        path: /webhook
    permissions:
      allow: [Read, Glob]  # call_agent is auto-added because agent-calls is non-empty
    runner:
      mode: docker
      docker:
        image: openaios/agent:latest
    capabilities:
      browser: false
      agent-calls: [coder]   # orchestrator may call coder

  - name: coder
    persona: "You are an expert software engineer."
    model:
      default: claude-code
    channels: {}             # No channel — only reachable via bus
    permissions:
      allow: [Read, Write, Bash, Glob, Grep]
    runner:
      mode: docker
      docker:
        image: openaios/agent:latest
    capabilities:
      browser: false
      agent-calls: []        # coder cannot call anyone

Bus HTTP server

PropertyValue
Bind address127.0.0.1 (loopback only, never public)
PortRandom ephemeral (bus_port: 0) or configured
EndpointPOST /internal/bus/message
AuthAuthorization: Bearer {randomUUID()} — one-time token generated at startup
Token deliveryInjected into containers via OPENAIOS_BUS_TOKEN env var at docker run time
URL deliveryInjected via OPENAIOS_BUS_URL env var

call_agent usage (inside container)

bash
# The call_agent tool is available in the PATH inside openaios/agent:latest
call_agent coder "Implement a binary search function in Python with tests"

# Returns the agent's response on stdout
# Exit code non-zero if the bus call fails or is rejected

Capabilities

Capabilities are optional features provisioned as sidecars or injected environment at startup. They require Docker mode unless noted.

browser

When capabilities.browser: true, a ghcr.io/zenika/alpine-chrome container is started alongside the agent container on the openaios Docker bridge network. The Chrome DevTools Protocol (CDP) endpoint is injected into the agent's environment:

bash
# /workspace/.env.capabilities inside the agent container:
CDP_URL=http://openaios-{agentName}-browser:9222

The agent can use the CDP URL to drive Chromium for web scraping, testing, or browser automation. Container name: openaios-{agentName}-browser. Reachable by DNS within the openaios network.

yaml
capabilities:
  browser: true    # Requires runner.mode: docker

agent-calls

Listing agents in capabilities.agent-calls does two things automatically:

  1. Adds call_agent to the agent's effective permissions.allow list (no manual entry needed)
  2. Registers the listed agents as the allowedCallees in the bus — calls to unlisted agents are rejected at the bus layer
yaml
capabilities:
  agent-calls: [coder, researcher]   # This agent may call 'coder' and 'researcher'

Budget & Governance

BudgetManager

The BudgetManager is backed by SQLite and records cost per agent per period. Before each turn, budget.check() is called. After each turn, budget.record() persists the cost. Agent-to-agent bus calls are budgeted against the callee agent's limit, not the caller's.

on_exceeded valueBehavior
blockRefuse to run the agent until the period resets.
downgradeSwitch to downgrade_to model for the rest of the period.
warnLog a warning but continue running. No enforcement.
yaml
budget:
  period: monthly
  agents:
    assistant:
      limit: 10.00           # $10 USD per month
      warning_at: 0.80       # Warn at $8.00
      on_exceeded: downgrade
      downgrade_to: ollama/qwen2.5:7b

    power-agent:
      limit: 50.00
      warning_at: 0.90
      on_exceeded: block

LocalGovernance

LocalGovernance is an in-process policy engine. It checks each tool call against the agent's permissions.allow and permissions.deny lists. The decision is deterministic and never fails — no network dependency.

  • Empty allow list → all tools blocked
  • * in allow → all tools permitted (use with caution)
  • deny list → takes precedence over allow

BRGovernance

BRGovernance calls an external policy API (the Bot Resources service) for each tool-use event. It can be configured to fail-open (allow if the API is unreachable, the default) or fail-secure (block if unreachable).

yaml
governance:
  br:
    url: ${BR_URL}
    token: ${BR_TOKEN}
    fail_secure: false   # false = allow on API timeout (default)
                         # true  = block on API timeout (strict mode)

LocalGovernance is always active. BRGovernance is an additional layer — both must permit the action for it to proceed.

Security

openAIOS is designed around the assumption that AI agent output is untrusted and that agents must not be able to escalate beyond their declared capabilities.

Threat model

ThreatMitigation
Prompt injectionDeny-by-default tools; -- separator prevents flag injection; input validation
Privilege escalationDocker isolation; no host FS mounts; memory/CPU limits
Credential exposureEnv var references only in config; .env chmod 600; systemd EnvironmentFile
Budget abusePer-agent spending limits with block/downgrade/warn enforcement
Unauthorized agent callsTwo-layer auth (governance policy + allowedCallees)

Key controls

1. Deny-by-default permissions

No tools are allowed unless explicitly listed. An empty allow list blocks everything.

yaml
permissions:
  allow: [Read, Glob]  # Only these two tools are allowed
  deny: [Bash]         # Explicit deny — overrides allow

2. CLI flag injection prevention

Both ClaudeCodeRunner and DockerRunner use spawn() (not shell: true) and always pass -- before user input, so user-supplied text can never be interpreted as a CLI flag.

typescript
spawn('claude', [...flags, '--', userMessage], { shell: false })

3. Input validation

InputValidation
Agent name/^[a-z0-9-]+$/
User / session ID/^[a-zA-Z0-9:_-]+$/
Message size≤ 16KB

4. Tailscale-first networking

Default network.bind: tailscale — the runtime listens only on the Tailscale interface. No public ports exposed. Telegram/Discord use outbound polling only. The bus HTTP server binds to 127.0.0.1 exclusively.

5. Docker isolation

In Docker mode: configurable memory/CPU hard limits, no host filesystem mounts (workspace via named Docker volume only), containers on the openaios bridge — not the host network.

6. Bus token auth

A one-time UUID bearer token is generated at startup and injected into containers via OPENAIOS_BUS_TOKEN. Requests without the correct token return 401.

7. Systemd hardening

ini
NoNewPrivileges=yes
PrivateTmp=yes
ProtectKernelTunables=yes
ProtectKernelModules=yes
ProtectControlGroups=yes
RestrictRealtime=yes
RestrictSUIDSGID=yes

Security checklist

  • Deny-by-default permissions in config schema
  • -- end-of-flags separator before all user input to claude CLI
  • Input validation: agent name regex, session ID regex, 16KB message limit
  • No credentials in config files — env vars only
  • Tailscale bind by default
  • Bus HTTP server bound to 127.0.0.1, one-time UUID token
  • Two-layer agent-to-agent authorization (governance + allowedCallees)
  • Docker container memory/CPU limits
  • Budget limits with block/downgrade/warn enforcement
  • Fail-open governance with explicit opt-in to fail-secure
  • Per-user rate limiting (future)
  • Rootless Docker / Docker socket proxy (future)
  • Audit log to file (future)

Known limitations

  • Native mode: agent isolation is user-level — all agents run as the aios user. Use Docker mode for stronger isolation.
  • In-memory history: Ollama and OpenAI-compat runners store conversation history in memory, lost on restart.
  • No per-user rate limiting — a malicious user could send many messages rapidly.
  • Docker socket access — Docker mode requires the aios user to be in the docker group, which is equivalent to root access. Consider rootless Docker.

Deployment

Production target: forge-smith (mini PC, Ubuntu 24.04, systemd). Transport: Tailscale SSH.

First-time server setup

Run once on the server as root:

bash
git clone https://github.com/NeoForgeEngineering/openaios /tmp/openaios-setup
sudo bash /tmp/openaios-setup/scripts/setup-systemd.sh

This creates:

  • aios system user at /home/aios/
  • Directory structure: /home/aios/openaios/, /home/aios/workspaces/, /home/aios/data/
  • Env file: /home/aios/.env (chmod 600 — edit with your secrets)
  • Systemd unit: /etc/systemd/system/openaios.service
  • Sudo rule: aios user can restart its own service

Deploy from Mac

bash
# Build locally
pnpm install && pnpm build

# Deploy (rsync + SSH restart via Tailscale)
REMOTE_HOST=forge-smith ./scripts/deploy.sh

# Dry run first
./scripts/deploy.sh --dry-run

What deploy.sh does:

  1. Verifies Tailscale SSH connectivity to forge-smith
  2. rsync built files to forge-smith (excludes .env, openAIOS.yml, data/, workspaces/)
  3. Runs pnpm install --prod on the remote
  4. Runs sudo systemctl restart openaios
  5. Verifies the service is running

Managing the service

bash
# Edit config (on forge-smith)
nano /home/aios/openaios/openAIOS.yml

# Edit secrets
nano /home/aios/.env

# Restart after config change
sudo systemctl restart openaios

# Follow logs
sudo journalctl -u openaios -f

# Last 100 lines
sudo journalctl -u openaios -n 100 --no-pager

GitHub Actions CI/CD

Push to main triggers automatic deploy via .github/workflows/deploy-forge-smith.yml.

Required GitHub secrets (in the forge-smith environment):

SecretDescription
FORGE_SMITH_HOSTTailscale hostname or IP of forge-smith
FORGE_SMITH_SSH_KEYPrivate SSH key for aios@forge-smith
TAILSCALE_OAUTH_CLIENT_IDFor GitHub Actions Tailscale connection
TAILSCALE_OAUTH_SECRETPaired with the OAuth client ID

Node.js management

Node.js 22 is managed via nvm for the aios user. The systemd unit includes the nvm path in Environment=PATH=. To upgrade: install the new version via nvm as the aios user, then update the path in the systemd unit.

Credential handling

bash
# /home/aios/.env (chmod 600, owned by aios)
TELEGRAM_TOKEN=...
ANTHROPIC_API_KEY=...
BR_URL=...
BR_TOKEN=...

Config files (openAIOS.yml) reference secrets via ${ENV_VAR} — never inline credentials. .gitignore excludes .env, openAIOS.yml, data/, and workspaces/.

Package Structure

openAIOS is a pnpm monorepo built with Turbo. Packages are layered — each depends only on packages below it in the dependency graph.

PackageNamePurpose
packages/core@openaios/coreZod schemas, TypeScript interfaces, config loader. Foundation for all other packages.
packages/runner@openaios/runnerRunner adapters: ClaudeCodeRunner, OllamaRunner, OpenAICompatRunner, DockerRunner. ContainerOrchestrator, CapabilityProvisioner.
packages/budget@openaios/budgetSQLite-backed BudgetManager — records costs, enforces period limits.
packages/governance@openaios/governanceLocalGovernance (in-process) and BRGovernance (external API) policy adapters.
packages/router@openaios/routerRouterCore (message pipeline), AgentBus, SessionStore.
packages/channels@openaios/channelsTelegramAdapter, WebhookAdapter, DiscordAdapter.
packages/cli@openaios/cliopenaios start/status/init commands. Wires all packages together.
packages/br-sdk@openaios/br-sdkBot Resources API client used by BRGovernance.
packages/br-hook@openaios/br-hookClaude Code hook scripts for BR integration.
packages/images/agent—Dockerfile: node:22 + claude CLI + call_agent bash tool. Base image for Docker-mode agents.
